## 一、从代码反推的 V1 功能设计

### 1. 项目定位与运行架构

当前 V1 是 Unity 客户端内的单机模拟经营 + 自动战斗 MVP。

代码采用“纯 C# 核心 + Unity 表现层”的分层方式：

| 层级 | 主要内容 | 关键文件 |
|---|---|---|
| 配置层 | JSON 配置加载、校验、配置数据库 | `ConfigLoader.cs`、`ConfigValidator.cs`、`JsonConfigSource.cs` |
| 运行时状态 | 国家、区域、据点、军队、外交、事件、命令队列、随机数 | `WorldRuntime.cs`、`SettlementState.cs`、`RegionState.cs` |
| 核心引擎 | 月度 Tick、命令队列、系统调度、事件发布 | `GameEngine.cs`、`SystemManager.cs`、`EventBus.cs` |
| 规则系统 | 人口、经济、建筑、征兵、行军、战斗、外交、AI | `Assets/_Game/Scripts/Systems` |
| 存档系统 | JSON 快照、3 个槽位、文件存储、读档恢复 | `SaveManager.cs`、`SaveConverter.cs`、`SaveData.cs` |
| Unity 表现层 | 启动、HUD、双层地图、选择、据点面板、摄像机 | `Presentation/*` |

`GameApplication` 是应用层门面：创建新世界、读档、驱动 `GameEngine`，并按真实时间累计触发月度 Tick。默认 `5` 秒等于 1 个游戏月，HUD 支持 1x、2x、3x 速度。

### 3. 月度 Tick 流程

游戏不是传统回合制界面，而是实时累计时间后自动结算月度 Tick。

实际执行顺序：

1. 时间推进 1 个月。
2. 执行上月命令队列：建设、征兵、行军、攻击、外交等命令先校验再执行。
3. 按注册顺序执行系统：
   - `PopulationSystem`：人口增长、民心恢复、粮荒后果。
   - `EconomySystem`：粮税、钱税、军粮、军饷。
   - `ConstructionSystem`：建造队列递减，完成后添加或升级建筑。
   - `RecruitSystem`：征兵队列递减，完成后加入守军。
   - `ArmySystem`：野战军行军、撤退、解散返还。
   - `BattleSystem`：围城状态自动结算战斗。
   - `DiplomacySystem`：根据外交关系刷新战争状态。
   - `AISystem`：非玩家、非中立势力生成下月命令。
4. 发布 `MonthChanged` 事件。
5. `GameApplication.Update()` 在 Tick 后执行自动存档，默认保存到槽位 1。

重要行为：AI 生成的命令不会当月立即执行，而是进入命令队列，在下一次 Tick 的命令阶段执行。

### 4. 人口、经济、民心

基础经济参数与配置：

| 参数 | 当前代码值 |
|---|---:|
| 每户人口 | 5 |
| 每户土地 | 50 亩 |
| 每亩年产粮 | 100 斤 |
| 粮税率 | 15% |
| 每人年钱税 | 2 铜钱 |
| 士兵月粮耗 | 90 斤 |
| 士兵月军饷 | 1 铜钱 |

每月经济按据点独立结算，没有把据点粮钱汇入国家国库。税收受建筑与民心影响：

| 民心 | 税收倍率 |
|---|---:|
| 80 及以上 | 100% |
| 60-79 | 80% |
| 40-59 | 50% |
| 40 以下 | 50% |

人口系统规则：

| 场景 | 当前实现 |
|---|---|
| 正常状态 | 民心每月 +1，人口按 0.2% 基础月增长，受民心倍率影响 |
| 人口上限 | 初始等于模板人口，民居可增加上限 |
| 粮荒 | 粮食为 0 时，民心 -5、守军逃散 5%、人口减少 1% |
| 土地 | 每月按户数重算，户数 = 人口 / 5，土地 = 户数 * 50 |

### 5. 建筑与征兵

建筑由 `Buildings.json` 配置，建设通过命令提交，先扣钱，进入建造队列，若已有同类建筑则完成后升级。

| 建筑 | 范围 | 成本 | 工期 | 效果 |
|---|---|---:|---:|---|
| 农田 | 村庄 | 100 | 2 月 | 粮税 +10% / 级 |
| 民居 | 村庄 | 150 | 2 月 | 人口上限 +20 / 级 |
| 粮仓 | 村庄 | 80 | 1 月 | 配置存在，当前未见直接容量约束逻辑 |
| 训练场 | 村庄 | 150 | 2 月 | 缩短征兵训练期 |
| 木墙 | 村庄 | 200 | 3 月 | 防御 +20% / 级 |
| 集市 | 城市 | 500 | 3 月 | 钱税 +20% / 级 |
| 官署 | 城市 | 400 | 3 月 | 配置存在，当前未见治理逻辑消费 |
| 兵营 | 城市 | 600 | 4 月 | 配置存在，当前未见驻军上限逻辑消费 |
| 石墙 | 城市 | 800 | 5 月 | 城防 +50% / 级 |

征兵通过 `RecruitCommand`：

| 条目 | 当前实现 |
|---|---|
| 征兵成本 | 每人 10 钱 + 100 斤粮 |
| 训练时间 | 默认 2 个月 |
| 训练场 | 每 2 级减少 1 个月，最少 1 个月 |
| 人口校验 | 人口至少达到征兵数量 / 10 |
| 完成效果 | 到期后直接加入该据点守军 |

### 6. 军事、行军、战斗

V1 只有一种士兵。平时以据点守军存在，出征时从据点抽调生成野战军。

出征限制：

| 参数 | 当前实现 |
|---|---:|
| 每势力最大野战军 | 3 |
| 单次最多抽调 | 当前守军 50% |
| 村最低留守 | 10 |
| 城最低留守 | 30 |
| 国都最低留守 | 50 |

行军规则：

- 只能从出兵据点所在区域前往相邻区域，或同区域内攻击。
- 野战军每月移动到目标相邻区域；到达后若有目标据点则进入 `Sieging`。
- 战斗失败或撤退时，幸存士兵返回来源据点；战斗胜利后幸存攻方转为目标据点守军。

战斗规则：

| 战力比 | 胜率 | 攻方损失 |
|---|---:|---:|
| > 2.0 | 100% | 10%-20% |
| 1.2-2.0 | 80% | 20%-40% |
| 0.8-1.2 | 50% | 30%-50% |
| < 0.8 | 20% | 40%-80% |

防御战力 = 守军 * 防御系数 * (1 + 防御建筑加成)。攻方胜利时防守方全灭；攻方失败时防守方损失 25%。

占领规则：

- 攻占单个据点后，该据点归属变更，民心设为 60。
- 区域归属不是攻下一个据点就变更，而是当区域内核心城和全部村庄都归属同一新势力时才变更。
- 区域占领会发布 `RegionCaptured` 事件。

### 7. 外交与 AI

外交关系为两两势力的数值，范围 -100 到 100，默认 0。关系档位：

| 关系值 | 档位 |
|---|---|
| >= 50 | Friendly |
| 0-49 | Neutral |
| -1 到 -49 | Hostile |
| -50 到 -79 | WarPossible |
| <= -80 | War |

AI 每月只处理秦、晋、齐、楚、周，不处理玩家和中立。AI 规则：

1. 计算粮食危机：势力所有据点粮食总量 < 守军月粮耗 * 6，则进入 `Weak`，停止本月决策。
2. 补守军：村目标 20、城目标 50、国都目标 100。每月最多对一个据点提交一条征兵命令。
3. 发展建设：据点铜钱大于 500 时，村庄优先农田，人口接近上限时造民居；城市造集市。每月最多提交一条建设命令。
4. 战争判断：只考虑相邻区域的非中立、非流民阶段玩家势力；关系 <= -70 且本国国力 >= 对方国力 * 1.3 时，有 50% 概率提交宣战命令。

国力计算：

```
人口 + 士兵 * 5 + 区域数 * 500 + 粮食 / 1000 + 铜钱 / 10
```

玩家威胁值函数已存在：

```
玩家城市数 * 30 + 玩家士兵 / 10
```

但该威胁值当前未接入 AI 行为。

### 8. 存档系统

存档采用 JSON 快照，当前用文件系统实现，槽位 1-3。

保存内容包括：

| 数据 | 当前实现 |
|---|---|
| 版本与摘要 | version、slot、显示名、年月 |
| 时间 | 年、月 |
| 国家 | 国库、AI 状态、战争状态 |
| 区域 | 归属、边境标记、城、村、邻接 |
| 据点 | 人口、户数、土地、粮钱、民心、守军、建筑、建设队列、征兵队列 |
| 野战军 | 所属、来源、当前位置、目标、兵力、士气、状态、行军进度 |
| 外交 | 两两关系 |
| 命令队列 | 尚未执行的命令 |
| 事件状态 | 进行中事件列表 |
| 随机数 | xorshift 状态，支持确定性恢复 |

自动存档时机：每次月度 Tick 后保存到槽位 1。手动保存、读档、退出保存已接入 HUD/菜单。攻城成功、国家灭亡未见单独即时存档触发。

### 8. Tick 流程

| 步骤 | MVP 核心设计 | 当前代码 |
|---|---|---|
| 经济与人口 | 农业生产、税收、军费，再人口增长 | 人口系统先执行，经济系统后执行 |
| 建筑完成 | 在人口后、民心后 | 经济后执行 |
| AI 与执行 | AI 决策后征兵与建设 | AI 只生成命令，下月执行 |
| 行军与战争 | AI 后执行 | 行军、战斗在 AI 前执行 |
| 自动存档 | Tick 最后 | `GameApplication.Update()` Tick 后执行 |

这是一处关键架构差异：代码采用命令队列把“决策”和“执行”隔离，导致 AI 和玩家本月提交的命令都延后一 Tick 生效。

## 三、差异结论

### 已高度符合 MVP 核心设计的部分

- 核心循环的主要系统已具备：人口、税收、建设、征兵、出征、自动战斗、占领、存档。
- 玩家初始村数值完全一致。
- 基础经济参数、士兵消耗、守军抽调限制、最低留守、战斗胜率区间基本一致。
- 3 槽 JSON 快照存档、每月自动存档、手动保存、退出保存均已实现。

### 主要偏差

- 世界规模少 1 座中立城，导致当前为 21 城/89 据点，而 MVP 核心设计为 22 城/90 据点。
- 若干模板数值与 MVP 核心设计不同，尤其普通村/城铜钱和国都粮钱。
- Tick 顺序与设计文档不同，并且命令采用“下月执行”模型。
- 民心恢复为 +1/月，而设计为 +2/月；民心低于 40 未实现叛乱。
- AI 只实现了基础发展和战争触发，高级威胁、紧张粮食、绝对优势战争、分级响应均未完成。
- 胜利目标、国家灭亡、控制洛邑问鼎天下等终局逻辑未实现。

### 代码已扩展但 MVP 核心设计未明确的部分

- 人口上限与粮荒后果链。
- 建筑工期和建筑升级。
- 征兵资源成本和训练期。
- 区域必须全据点占领后才改变归属。
- 命令队列、事件总线、确定性随机数持久化。
- 双层地图、HUD、速度控制、暂停、基础点击操作与摄像机控制。
