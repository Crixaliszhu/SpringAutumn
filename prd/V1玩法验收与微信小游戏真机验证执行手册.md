# 《春秋问鼎》V1 玩法验收与微信小游戏真机验证执行手册

> 生成日期：2026-06-24
> 适用项目：`/Users/ypmma106/AI/SpringAutumn`
> 当前工程：Unity/Tuanjie `2022.3.62t10`，Tuanjie Editor `1.9.2`
> 当前包特征：已包含 `cn.tuanjie.codely.bridge`

本文档用于把 V1 从“Editor 内流程跑通”推进到“手机上可验证核心循环”。目标不是一次性达到发布品质，而是尽早发现手机端、微信小游戏端、输入、性能、存档和节奏问题。

---

## 一、总原则

### 1. 当前阶段最重要的目标

V1 当前已经跑通核心流程。下一阶段的优先级是：

1. 完成 V1 玩法验收，确认核心循环在 Editor/本地环境中稳定。
2. 冻结一个可验证版本，不继续扩展 V1.1 功能。
3. 走通微信小游戏构建与真机验证链路。
4. 在手机上验证“能不能玩、看不看得清、点不点得准、卡不卡、存不存得住”。

### 2. 不在本阶段追求的内容

以下内容除非阻断 V1 核心体验，否则放入 V1.1：

- 完整外交系统。
- 叛乱系统。
- 将领系统。
- 国家灭亡演出。
- 完整胜利动画。
- 更复杂 AI 策略。
- 大规模美术、音效、商业化、登录、排行榜。

### 3. 阶段通过定义

V1 进入微信真机验证前，必须满足：

- 新开局无启动错误。
- 玩家能在一局中完成：经营、建设、征兵、出兵、攻占至少 1 个非玩家据点。
- 存档、读档后，年月、资源、据点归属、建设队列、征兵队列、军队状态不丢失。
- 没有阻断核心流程的崩溃、卡死、无法点击、无法保存问题。

### 6. V1 问题分级

| 级别 | 定义 | 处理方式 |
|---|---|---|
| P0 阻断 | 启动失败、崩溃、无法保存/读档、核心操作完全不可用 | 必须立即修复，不能进入下一阶段 |
| P1 必修 | 玩家无法顺利完成核心循环，或 UI 明显误导，或数值导致死锁 | V1 真机验证前修复 |
| P2 可调 | 节奏不舒服、数值偏慢/偏快、AI 太弱/太强，但流程可完成 | 可在真机验证前后集中调优 |
| P3 延后 | 不影响 V1 核心验证的系统缺口或体验增强 | 进入 V1.1 backlog |

### 7. 当前建议重点关注项

根据当前代码与设计梳理，V1 试玩时优先关注：

- 玩家进攻当前固定从 `V_PLAYER_001` 出兵，扩张后是否影响体验。
- 外交按钮目前只是提示“后续接入”，是否应隐藏或改文案，避免误解为 bug。
- 民心恢复代码为每月 +1，设计文档为 +2；试玩时确认节奏是否可接受。
- 中立势力当前没有中立城，是否影响“统一边境郡”的目标感。
- 胜利目标、国家灭亡、控制洛邑尚未形成闭环，是否至少需要 V1 提示目标。

## 四、微信小游戏流程概览

### 1. 推荐路线

当前工程使用 Tuanjie Editor，且已有 `cn.tuanjie.codely.bridge`。优先路线如下：

1. 使用 Tuanjie/Unity 当前工程完成本地 V1 验收。
2. 检查是否已安装并启用微信小游戏相关构建/转换能力。
3. 从 Editor 构建微信小游戏工程。
4. 用微信开发者工具打开导出的小游戏工程。
5. 在微信开发者工具内完成模拟器启动。
6. 使用真机预览/真机调试验证手机体验。

备选路线：

- 若当前 Tuanjie 环境无法直接导出微信小游戏，则再考虑传统 Unity WebGL 构建 + 微信小游戏 WebGL 转换工具链。
- 备选路线只用于打通验证，不建议同时维护两套正式流程。

### 2. 阶段目标

| 阶段 | 目标 | 通过标准 |
|---|---|---|
| W0 环境确认 | 确认 Editor、包、微信开发者工具、AppID/测试号可用 | 能看到小游戏相关构建入口或明确替代路线 |
| W1 构建导出 | 从当前工程生成微信小游戏工程 | 导出目录完整，无构建失败 |
| W2 开发者工具启动 | 微信开发者工具打开项目 | 模拟器能启动到游戏首屏 |
| W3 真机打开 | 手机扫码预览/调试 | 能进入游戏并看到 HUD/地图 |
| W4 核心循环 | 手机上完成一次最小核心循环 | 建设、征兵、出兵、占领、存读档至少跑通 |
| W5 记录结论 | 汇总平台问题 | 明确 P0/P1/P2/P3 和下一步 |

## 五、微信小游戏迁移前检查

### 1. 工程检查

| 检查项 | 当前情况 | 风险 | 处理建议 |
|---|---|---|---|
| Editor 版本 | `2022.3.62t10` / Tuanjie `1.9.2` | 需使用对应导出链路 | 优先使用 Tuanjie 微信小游戏能力 |
| 微信桥接包 | `cn.tuanjie.codely.bridge` 已存在 | 需要确认实际构建入口 | 在 Package Manager/菜单检查 |
| 配置加载 | 从 `Application.dataPath/_Game/Config` 读取 JSON 文件 | 小游戏/WebGL 文件路径可能不同 | 手机验证前重点检查配置是否能加载 |
| 存档 | `Application.persistentDataPath/saves/save_*.json` | 小游戏沙盒文件系统可能表现不同 | 必须真机验证保存/读取 |
| 输入 | 鼠标与单指触控适配 | 手机点击精度、拖动、缩放需验证 | 真机重点测试 |
| 字体 | TextMeshPro + CJK fallback | 小游戏包体和字体加载风险 | 真机看中文是否缺字/模糊 |
| 性能 | 24 区域/89 据点 | 手机 WebGL/小游戏性能不确定 | 测 FPS、发热、加载时间 |

### 2. 资源与包体检查

| 检查项 | 要求 |
|---|---|
| 未使用资源 | 尽量剔除示例资源、无用素材 |
| TextMeshPro 字体 | 确认中文字体资源不会过大，且真机显示正常 |
| 场景数量 | 只纳入 V1 必需场景 |
| 配置 JSON | 确保被包含到构建产物或改入 Resources/StreamingAssets 可访问路径 |
| 日志 | 保留关键启动/存档错误日志，避免大量刷屏 |

### 3. 代码兼容重点

| 代码点 | 当前文件 | 验证重点 |
|---|---|---|
| 配置路径 | `GameLauncher.Initialize()` | `Application.dataPath` 在小游戏环境下是否能读取 `_Game/Config` |
| 存档路径 | `FileSaveStorage` | `persistentDataPath` 是否可写，3 槽是否稳定 |
| JSON 序列化 | `DataContractJsonSerializer` | IL2CPP/WebGL/小游戏环境是否兼容 |
| 反射/泛型 | `EventBus`、序列化 | AOT 环境是否有裁剪或运行异常 |
| 触控 | `TouchInputAdapter`、`InputManager` | 手机上点击、拖动、UI 阻挡是否正确 |

## 八、常见风险与处理策略

### 1. 配置加载失败

症状：

- 启动白屏。
- Console 报找不到 `Nations.json`、`Regions.json` 等。
- 游戏世界未创建。

优先处理：

- 确认 JSON 配置是否进入小游戏构建产物。
- 若 `Application.dataPath/_Game/Config` 在小游戏不可用，改为可跨平台访问的资源加载方式，例如 `Resources`、`StreamingAssets` 或微信插件推荐路径。
- 保持 `IConfigSource` 接口不变，只替换表现层配置源。

### 2. 存档失败

症状：

- 保存按钮显示失败。
- 读档后为空。
- 关闭小游戏后存档丢失。

优先处理：

- 确认 `Application.persistentDataPath` 在小游戏环境下是否可写。
- 若 `FileSaveStorage` 不适配，新增微信小游戏专用 `ISaveStorage` 实现。
- 保持 `SaveManager` 和 `SaveConverter` 不变，只替换底层存储。

### 3. 触控不顺

症状：

- 地图拖动和点击冲突。
- 点击区域误选。
- 面板按钮太小。

优先处理：

- 调整 `clickMoveThreshold`、`clickTimeThreshold`。
- 增大按钮和可点击碰撞区域。
- 区分 UI 点击与地图点击，避免穿透。

### 4. UI 适配问题

症状：

- HUD 被裁切。
- 据点面板遮挡地图。
- 攻击确认输入框太小。

优先处理：

- 先选定 V1 手机方向：横屏或竖屏。
- 固定安全区和锚点策略。
- 优先保证 HUD、据点面板、攻击确认面板可用。

### 5. 性能/加载问题

症状：

- 首次加载很慢。
- 地图切换卡顿。
- 运行一段时间后掉帧。

优先处理：

- 减少首包无用资源。
- 关闭非必要日志。
- 检查地图视图是否重复创建对象。
- 优先保证 24 区域/89 据点规模下稳定。
